### Cost note: API calls are gated by wall-clock, not the game clock

The live viewer uses the non-blocking `CivilizationDecisionScheduler`, which
launches at most one model call every ~6 real seconds, independent of how many
simulated hours elapse. In ~10 minutes that is ~75 calls regardless of sim
speed; the deterministic `FallbackGovernor` reuses the last decision for the
intervening hours. This is why a 43-in-game-day watched run costs about the same
as the headless 96-hour PR #29 gate, which made ~96 calls: cost tracks the
number of API requests (~75 vs ~96), not the simulated duration.

Caveat on telemetry: the decision log records one row per sim-hour and carries
the last completed call's `llm_latency` forward, so a naive count of
`llm_latency > 0` rows (1031) overstates the call count. The true distinct-call
count (~75) comes from counting transitions into the `thinking` state / changes
in `llm_latency`. The run log still does not capture provider token usage or
cost; any dollar figure is user-reported from the OpenAI billing surface.

## Observation

The infrastructure path is healthy: hosted `gpt-5.4-mini` authenticated,
returned valid structured output, governed the live viewer with zero dropped
decisions, and the fallback covered every non-call hour.

The colony, however, collapsed under the model's management and never recovered.
Despite a decision every cooldown cycle, the governor let bread deplete on day 6
and water stall on day 16, then sat in a chronic low-mood (~40-45) plateau for
the rest of the run. The applied-action mix is the notable signal: the model
spent the overwhelming majority of its effort churning `set_schedule` (6060
applications) while `set_work_priority` (408) and the actual supply problem went
under-corrected. Staffing was not the bottleneck - farming had 4 pawns and the
water well had 1 throughout - yet food and water both reached 0.0. This points
at governor decision quality (over-favoring `set_schedule`), not a runtime bug,
as the next thing to investigate.
